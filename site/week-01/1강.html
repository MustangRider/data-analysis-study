1강 — 왜 리뷰를 분석하는가 & AI 앱 개발자란

🎯 학습 목표

  • 리뷰를 사람이 직접 읽지 않고 AI로 분석해야 하는 이유를 설명할 수 있다
  • 우리가 되려는 "AI 앱 개발자"가 AI 모델 개발자와 어떻게 다른지 구분할 수 있다
  • 6일 동안 결국 무엇을 만들게 되는지 전체 목표를 이해할 수 있다

📖 해설

1. 출발점 — 리뷰 10만 개를 어떻게 보시겠어요?

여러분이 게임 회사의 담당자라고 해봅시다. 우리 게임에 리뷰가 10만 개 쌓였어요. 유저가 뭘 좋아하고 싫어하는지 전부 그 안에 들어 있습니다. 그런데 이걸 어떻게 보시겠어요? - 팀원들과 나눠 읽는다? — 시간이 감당이 안 됩니다. - 별점만 본다? — 왜 그런 별점이 나왔는지 이유를 알 수 없어요. - 최신 것만 본다? — 트렌드도, 그 게임의 특장점도 놓치게 됩니다.

데이터(리뷰)는 분명히 있는데, 사람의 방식으로는 의미를 못 뽑는 상황이죠. 바로 여기서 AI 분석이 등장합니다.

리뷰를 제대로 분석하면 네 가지를 얻습니다. - 유저 인사이트 — 유저가 무엇을 좋아하고 싫어하는지 직접 듣습니다. - 빠른 대응 — 이슈가 터졌을 때 실시간으로 여론을 파악해 바로 움직일 수 있습니다. - 경쟁 분석 — 우리 게임뿐 아니라 경쟁 타이틀의 리뷰도 똑같은 방식으로 분석할 수 있습니다. - 트렌드 추적 — 시간에 따른 반응 변화를 데이터로 확인합니다.

2. 두 개의 실제 사례

실제로 이런 일이 있었습니다.

성공한 경우. 한 게임 회사가 대규모 업데이트(새 콘텐츠 + 밸런스 조정)를 했는데, 별점이 하루 만에 4.5에서 3.2로 폭락했어요. 리뷰는 하루 수천 개씩 쏟아지고, 팀원들이 밤새 읽었지만 사람마다 읽은 리뷰가 달라서 결론이 제각각이었습니다(밸런스다 / 버그다 / UI다). 그래서 AI로 2천 개 리뷰를 전부 긍·부정으로 분류하고, 부정 리뷰의 키워드 빈도를 분석했더니 30분 만에 1위 키워드 "매칭"이 나왔어요. 새 매칭 시스템이 기존 유저에게 너무 불리하게 바뀐 게 원인이었죠.

실패한 경우. AI를 쓰기 전에는 인턴 세 명이 한 달 동안 리뷰를 분류했습니다. 그런데 기준이 제각각이었어요. 같은 리뷰를 한 명은 중립, 한 명은 부정, 또 한 명은 아예 다른 키워드 체계로 분류했죠. 결국 한 달치 데이터를 통째로 못 쓰게 됐습니다. 여기서 꼭 기억하세요. 사람이 대량의 데이터를 일관되게 분류하는 건 구조적으로 불가능합니다. 의지나 능력의 문제가 아니라, 사람이라면 누구나 부딪히는 한계예요.

3. 사람의 한계 3가지, 그리고 AI

사람이 리뷰를 직접 읽을 때는 세 가지 벽에 부딪힙니다.

사람 AI
속도 리뷰 1건 판단에 30초~1분. 하루 100~200개가 한계 1만 개도 몇 분~몇십 분. 잠도 휴일도 필요 없음
일관성 아침/저녁, 사람마다 기준이 흔들림 1번째든 10만 번째든 똑같은 기준
비용 인건비·전문업체 비용 압도적으로 저렴 (단, 공짜는 아님 ↓)

⚠️ AI도 공짜는 아닙니다. GPT·Claude 같은 LLM을 우리 서비스에 붙일 때는 API라는 방식을 쓰는데 보통 유료예요(리뷰 1건당 몇 원 수준). 이 과정 뒤쪽에서 무료로 우수한 LLM을 쓰는 팁도 알려드릴게요.

4. 가장 중요한 원칙 — AI는 분류, 사람은 해석

방금 성공 사례에서 AI가 한 일은 "부정 1위 키워드는 매칭"을 찾아낸 것까지입니다. "그래서 매칭 시스템을 어떻게 되돌릴까"를 결정한 건 사람이었어요.

AI는 무엇을 봐야 하는지를 찾아줍니다. 그걸 해석하고 결정하는 건 여러분의 몫이고요. 이게 AI 시대의 협업 방식입니다.

5. 그래서 6일 동안 뭘 만들까요?

우리가 만들 최종 결과물은 게임 리뷰 분석 대시보드입니다. 네 가지 기능으로 이뤄져요. 1. 감성 분석 — 리뷰를 긍정/부정/중립으로 자동 분류합니다. 왜 부정인지(버그 때문인지, 콘텐츠 부족 때문인지) 근거까지 봅니다. 2. 키워드 추출 — 수천 개에서 자주 나오는 걸 자동으로 뽑아요. 단순 빈도가 아니라 맥락을 이해해서, "재미없음"과 "재미가 없어요"를 같은 의미로 묶습니다. 3. 트렌드 분석 — 시간에 따른 변화를 봅니다(지난달 그래픽 칭찬 → 이번 달 버그 불만 급증). 4. 대시보드 — 위 결과를 그래프·차트·워드클라우드로 시각화해서, 비개발자 동료나 경영진도 바로 이해하게 만듭니다.

6일 흐름은 이렇습니다 — 1~2일 데이터 수집·정리 → 3~4일 LLM으로 텍스트 분석 → 5~6일 대시보드 제작.

6. "AI 앱 개발자"는 엔진이 아니라 자동차를 조립하는 사람입니다

비유를 하나 들어볼게요. - AI 모델 개발자는 자동차 엔진을 직접 설계하는 사람입니다. 석·박사급에, 딥러닝 이론·수학·논문 역량이 필요하고, 수도 적어요. - 우리가 될 AI 앱 개발자는 이미 공개된 강력한 엔진(GPT·Claude·Gemini·Grok)을 API로 가져다 원하는 자동차를 조립하는 사람입니다. 엔진을 직접 만들 필요가 없어요.

예전에는 감성 분석 모델을 직접 학습시키고, 알고리즘을 짜고, 서버를 세우는 데 몇 년이 걸렸습니다. 지금은 LLM API 하나로 감성분석·키워드추출·요약이 다 됩니다. 그래서 우리가 배울 건 ① API를 쓰는 법(금방 익혀요)과 ② 어떻게 물어볼지, 즉 프롬프트를 설계하는 법입니다. 좋은 질문이 좋은 답을 끌어내거든요.

7. 왜 지금 이 역할이 주목받을까요?

  1. LLM의 등장 — 예전엔 작업마다(감성분석·요약·번역) 다른 모델이 필요했어요. 이제는 LLM 하나로, 프롬프트만 바꾸면 다 됩니다.
  2. 민주화 — 예전엔 빅테크 안에서만 가능했던 걸, 이제 인터넷만 되면 누구나 API 키 발급 + 코드 몇 줄로 씁니다.
  3. 기업의 다급함 — 모두가 "AI로 뭔가 해야 한다"는 압박을 받아요. 그런데 모델 개발자는 드물고 비싸니, 있는 AI를 잘 활용해 서비스를 만들 사람의 수요가 큽니다.
  4. 기술 장벽 붕괴 — 전공이냐 비전공이냐, AI 지식이 있느냐는 더 이상 중요하지 않아요. 파이썬 기초 + API 호출 + 프롬프트면 충분합니다.

8. 이 과정에서 기대하셔도 좋은 것

  1. 동작하는 서비스 — 이론만 배우고 끝나는 게 아니라, 리뷰를 넣으면 진짜 분석되는 걸 여러분 손으로 만듭니다.
  2. 포트폴리오 — "AI 기반 리뷰 분석 대시보드"라는 그럴듯한 결과물이 생겨요. 면접에서 직접 설명할 수 있습니다.
  3. 실무 감각 — 튜토리얼은 항상 성공하지만, 실무에선 에러·예상 밖 결과·비용 문제가 터집니다. 그걸 회피하지 않고 직접 겪어볼 거예요.
  4. 현장 노하우 — 재시도 로직, 대량 데이터 배치 처리, 결과 캐싱(이전 결과 재활용)처럼 책엔 없지만 현장에 꼭 필요한 것들을 다룹니다.

9. 꼭 기억할 4대 원칙

  1. 작성은 AI가, 해석은 여러분이. 파이썬 문법을 달달 외울 필요는 없지만, 코드 흐름을 읽고 AI가 엉뚱한 짓을 하는지 검증할 수는 있어야 해요. 여러분은 코더가 아니라 코드의 품질을 판단하는 리뷰어입니다.
  2. 구현보다 설계. 복잡한 알고리즘은 AI가 순식간에 만들어줍니다. 중요한 건 어떤 기능을 어떤 순서로 짤지 설계하는 논리력과 창의성이에요. AI는 벽돌을 쌓는 인부이고, 설계도를 그리는 건축가는 여러분입니다.
  3. 에러 해결의 주도권은 여러분이. 무작정 "고쳐줘"만 하지 말고, 에러 메시지를 읽고 어디가 왜 틀렸는지 보는 눈을 기르세요. 고친 이유를 이해해야 진짜 내 것이 됩니다.
  4. AI는 기술, 여러분은 도메인. 코딩은 AI에 맡기고, 아낀 시간을 이 기능이 왜 비즈니스적으로 가치 있는지 고민하는 데 쓰세요. 도메인 지식과 비즈니스 통찰은 AI가 대체할 수 없습니다.

10. 시작 전, 가벼운 마음가짐

이 과정을 따라올 때 도움이 되는 태도 몇 가지입니다. - 프로그래밍은 기초면 충분합니다. Python 기본 문법 수준이면 충분히 따라올 수 있어요. 전공자가 아니어도 괜찮습니다. - 완벽보다 동작. 처음부터 깔끔한 코드를 노리지 말고, 일단 돌아가게 만든 뒤 다듬으세요. - 에러는 정상입니다. 막히는 건 실패가 아니라, 해결하는 과정 자체가 학습이에요. - 막히면 바로 질문. 혼자 오래 끙끙대지 말고 빠르게 물어보는 편이 훨씬 효율적입니다.

🎮 게임 기획자 시점

사실 이 강의 전체가 기획자가 일하는 방식에 관한 이야기입니다. 리뷰 분석은 기획 의사결정의 근거를 모으는 도구예요. AI가 "부정 1위는 매칭"이라고 좁혀주면, 밸런스를 되돌릴지·BM을 손볼지·콘텐츠 우선순위를 어디에 둘지 결정하는 건 기획자입니다. 4대 원칙 중 "AI는 기술, 여러분은 도메인"이야말로 게임 기획자가 이 분야에서 가장 강한 지점이에요 — 게임을 아는 사람이 AI를 부리는 겁니다.

⚠️ 흔한 오해 / 짚고 갈 점

  • "AI가 답을 준다"가 아닙니다. AI는 무엇을 볼지를 찾아줄 뿐, 결정은 사람이 합니다.
  • "코딩을 다 배워야 한다"도 아니에요. 흐름을 읽고 검증할 정도면 되고, 핵심은 무엇을 만들지 설계하는 능력입니다.

✅ 30초 정리

  1. 사람은 리뷰를 속도·일관성·비용 때문에 대량으로 분석할 수 없습니다 → AI가 분류하고 사람이 해석·결정합니다.
  2. 우리는 엔진(모델)을 만드는 게 아니라 있는 AI를 조립하는 "AI 앱 개발자"가 됩니다. 무기는 파이썬 기초 + API + 프롬프트 + 도메인 감각이에요.
  3. 6일 뒤 결과물은 게임 리뷰 분석 대시보드(감성분석·키워드·트렌드·시각화)입니다.
