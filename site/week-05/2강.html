2강 — 로컬 개발환경 설정

🎯 학습 목표

  • 왜 Colab에서 벗어나 내 컴퓨터(로컬)에서 개발해야 하는지 설명할 수 있다
  • Python 3.12 안정 버전을 직접 설치하고, "왜 최신 버전이 아닌가"를 이해할 수 있다
  • 가상환경(venv)이 무엇이고 왜 필요한지 이해하고, 생성·활성화·비활성화할 수 있다
  • VS Code·Cursor 같은 IDE와 Claude Code·Gemini CLI 같은 AI 코딩 도구가 어떤 역할을 하는지 안다

📖 해설

1. Colab은 편했는데, 왜 굳이 내 컴퓨터로 옮길까요?

지금까지 우리는 구글 Colab에서 코드를 실행했습니다. 브라우저만 열면 되고, 설치할 것도 없고, 무료 GPU까지 빌려 쓸 수 있으니 정말 편했죠. 학습하고 실험하는 용도로는 Colab이 최고입니다. 그런데 이제부터는 여러분 컴퓨터에서 직접 Python을 돌릴 겁니다. 왜 그래야 할까요?

한마디로 말씀드리면, "앱(서비스)을 만들어 남에게 제공하려면 Colab으로는 한계가 있기 때문"입니다. 우리의 최종 목표는 리뷰 분석 대시보드라는 실제 앱을 만들고 배포하는 것입니다. 그 단계로 가려면 Colab의 다음 네 가지 한계를 넘어야 합니다.

Colab의 한계 무슨 문제가 생기나
세션 끊김 90분이 지나면 자동 종료됩니다. 데이터가 날아가죠. 긴 작업을 하다 통째로 날려본 분도 계실 겁니다.
파일 관리 불편 우리가 만들 앱은 app.py, database.py, utils.py처럼 여러 파일로 구성됩니다. Colab은 노트북 하나 단위라 이런 구조를 만들기 어렵습니다.
항상 온라인 필요 인터넷이 없으면 코드 실행 자체가 안 됩니다.
배포 불가능 가장 중요한 점입니다. 만든 앱을 다른 사람에게 제공할 방법이 없습니다.

로컬 개발환경은 이걸 정반대로 뒤집습니다. 내 컴퓨터에서 실행하니 시간 제한이 없고, 파일을 자유롭게 여러 개로 구성할 수 있고, 오프라인에서도 돌아갑니다. 그리고 이 코드를 AWS·GCP 같은 클라우드에 올리면 배포까지 가능해집니다. 그러면 전 세계 누구나 우리가 만든 앱에 들어와 사용할 수 있게 되죠.

구분 Google Colab 로컬 개발환경
실행 위치 브라우저(클라우드) 내 컴퓨터
설치 필요 없음 Python 설치 필요
자원 무료 GPU 사용 가능 내 하드웨어 사용
시간 제한 세션 90분 제한 시간 제한 없음
파일 구조 제한적(노트북 1개) 자유로운 파일 구조
배포 불가능 가능

핵심을 한 줄로 정리하면 이렇습니다. 내 컴퓨터가 곧 나만의 서버가 되는 겁니다. Colab에서 코드를 실행하던 걸 이제 내 컴퓨터에서 시간 제한 없이 안정적으로 돌리고, 노트북 파일 하나였던 걸 여러 파일짜리 체계적인 프로젝트로 키우고, 학습·실험용이던 작업을 실제 서비스 개발과 배포로 넘어가게 만듭니다. 개발자로 성장하려면 로컬 개발환경 설정은 사실상 필수입니다.

2. Python 설치 — 함정이 하나 있습니다

"공식 사이트에서 그냥 다운받으면 되는 거 아닌가요?" 맞는 말 같지만, 여기에 함정이 하나 있습니다. python.org에 들어가면 화면 한가운데 큰 버튼으로 "최신 버전 다운로드"가 보입니다. 그 버튼을 누르시면 안 됩니다.

왜냐하면 최신 버전(예: 3.13 같은 신버전)은 우리가 쓸 라이브러리들과 호환성 문제가 생길 수 있기 때문입니다. 우리는 지금 시점에서 검증된 Python 3.12 버전을 설치할 겁니다. 설치 절차는 이렇습니다.

  1. python.org 접속 — https://www.python.org/downloads/ (메인 페이지의 큰 버튼은 누르지 않습니다)
  2. 특정 릴리스 찾기 — 페이지를 아래로 내려 "Looking for a specific release?" 영역의 "View the full list"를 클릭해 전체 릴리스 목록을 연 뒤, 그 안에서 3.12.x 버전을 찾습니다.
  3. 내 OS에 맞는 설치 파일 다운로드 — Windows라면 보통 64-bit 설치 파일을 받으면 됩니다. (다운로드한 파일을 실행하면 설치 창이 뜹니다.)
  4. "Add Python to PATH" 반드시 체크 — 설치 창 맨 아래에 있는 이 체크박스를 꼭 켠 뒤 "Install Now"를 누릅니다.

마지막 4번이 정말 중요합니다. "Add Python to PATH"를 체크하지 않으면, 나중에 터미널에서 python 명령을 칠 때 "그런 명령 없다"는 오류가 나거나, Python이 설치된 경로를 일일이 직접 지정해 줘야 해서 아주 번거로워집니다. 혹시 체크를 빠뜨리고 설치했다면, 설치 파일을 다시 실행해 이 체크박스를 켜고 재설치하면 됩니다.

PATH가 뭐냐고요? 쉽게 말해 "이 폴더들 안에 있는 프로그램은 어디서든 이름만 불러도 실행해라" 하고 운영체제에 알려주는 목록입니다. Python을 PATH에 추가하면, 어느 폴더에 있든 터미널에서 그냥 python이라고만 쳐도 Python이 실행됩니다.

왜 최신이 아니라 3.12인가요?

세 가지 이유가 있습니다.

  1. 라이브러리 호환성 — FastAPI, Streamlit 같은 패키지가 아직 최신 Python을 지원하지 않을 수 있습니다. 설치는 되는데 막상 실행하면 에러가 나는 경우가 생깁니다.
  2. 실무 관행 — 대부분의 회사가 최신보다 한두 단계 아래의 안정화 버전을 씁니다. 안정성이 검증된 버전을 선호하는 거죠.
  3. 문제 해결이 쉬움 — 3.12 관련 자료가 인터넷에 아주 많아서, 문제가 생겨도 검색하면 해결책을 금방 찾을 수 있습니다.

여기서 개발의 중요한 깨달음 하나를 짚고 갑니다. 개발에서는 "안정성"이 "최신"보다 중요합니다. 우리는 흔히 "최신이 무조건 좋다"고 오해하지만, 실제 개발에서는 검증된 버전을 더 가치 있게 봅니다. 비유하자면 갓 나온 신차보다, 결함이 발견돼 리콜까지 끝난 차가 더 안전한 것과 같습니다. 실무에서는 검증된 버전을 선호한다 — 기억해 두세요.

3. 설치가 잘 됐는지 확인하고, 작업 폴더 만들기

설치가 끝났으면 Windows에서 Windows 키를 누르고 cmd를 입력해 터미널 창(검은 화면)을 엽니다. 여기서 다음 명령으로 설치를 확인합니다.

# Python이 잘 깔렸는지 버전을 확인 (3.12.x 가 나오면 성공)
python --version

참고로 터미널에 python만 치고 엔터하면 Python 셸(shell) 모드로 들어갑니다. 여기서도 코드 한 줄씩은 실행되지만, 긴 코드를 작성하기엔 불편하니 우리는 이 모드로 작업하지 않습니다. 혹시 셸 모드에 들어갔다면 exit()를 입력하거나 Ctrl + Z를 누르고 엔터를 치면 빠져나올 수 있습니다.

이제 앞으로 코드를 작성할 작업 폴더를 하나 만들어 보겠습니다.

# 'AI-reviews' 라는 작업 폴더(디렉토리)를 만든다  (mkdir = make directory)
mkdir AI-reviews

# 만든 폴더 안으로 이동한다  (cd = change directory)
cd AI-reviews

여기서 작은 팁 하나. 폴더 이름이나 명령을 손으로 다 치면 오타가 나기 쉽습니다. 터미널이 익숙하지 않다면 Tab 키를 적극 활용하세요. 이름의 앞부분만 치고 Tab을 누르면 나머지가 자동 완성됩니다. 이 Tab 키 하나가 오타를 크게 줄여서 생산성을 많이 높여 줍니다. 앞으로는 바로 이 AI-reviews 폴더 안에 코드를 쌓아 나가겠습니다.

4. 가상환경(venv) — 프로젝트마다 따로 쓰는 도구 상자

Python을 설치했으면 바로 쓰면 될 것 같은데, 그렇게 하면 나중에 문제가 생깁니다. 왜냐하면 프로젝트마다 필요한 라이브러리 버전이 다를 수 있기 때문입니다.

예를 들어 봅시다. A 프로젝트에서는 pandas 1.5 버전이 필요하고, B 프로젝트에서는 pandas 2.0 버전이 필요하다고 합시다. 그런데 컴퓨터 전체에 Python 하나만 깔려 있으면, pandas는 둘 중 한 버전만 설치할 수 있습니다. 결국 둘이 충돌하죠. 우리는 이 AI 리뷰 분석 앱을 만들고 나면, 앞으로 또 다른 프로젝트, 또 다른 프로젝트를 계속 만들게 됩니다. 그때마다 라이브러리가 서로 충돌해서 "어떤 건 되는데 어떤 건 안 되는" 이상한 오류들이 쏟아질 겁니다.

그래서 가상환경을 씁니다. 가상환경은 "프로젝트별로 독립된 도구 상자"라고 생각하시면 편합니다. 각 프로젝트가 자기만의 독립된 환경을 가지므로, 그 프로젝트에 필요한 버전의 라이브러리를 따로 설치하고 관리할 수 있습니다. 서로 영향을 주지 않죠.

가상환경 만드는 명령은 모든 OS에서 똑같습니다. 다만 활성화하는 명령은 OS마다 다릅니다.

# [모든 OS 공통] 'venv' 라는 이름의 가상환경을 만든다
#   python -m venv  : Python으로 가상환경 도구를 실행
#   맨 뒤 venv       : 우리가 만드는 가상환경의 이름 (보통 venv 또는 env 를 많이 씀)
python -m venv venv

이 명령을 실행하면 현재 폴더 안에 venv라는 폴더가 새로 생깁니다. 이 폴더 안에 가상환경 설정이 들어 있죠. 그런데 만드는 것만으로는 끝이 아닙니다. 반드시 "활성화"를 해야 적용됩니다. 활성화하지 않은 상태에서도 python은 동작하지만, 그건 컴퓨터 전체에 깔린 Python(전역 Python)을 쓰는 것이라 프로젝트가 쌓일수록 충돌·에러가 다시 발생합니다.

# [Windows - PowerShell] 활성화  (이 강의의 기본 환경)
.\venv\Scripts\Activate.ps1

# [Windows - CMD(명령 프롬프트)] 활성화
venv\Scripts\activate

# [Mac / Linux] 활성화  (source 명령을 씁니다)
source venv/bin/activate

참고: Windows PowerShell에서 위 명령이 보안 정책 때문에 막힌다면, PowerShell을 관리자로 열고 Set-ExecutionPolicy -Scope CurrentUser RemoteSigned를 한 번 실행한 뒤 다시 활성화하면 됩니다.

활성화에 성공하면 터미널 줄 맨 앞에 (venv)라는 표시가 붙습니다.

# 활성화 성공 시 프롬프트 모습 (앞에 (venv) 가 붙음)
# Windows:   (venv) C:\Users\...>
# Mac/Linux: (venv) user@mac %

이 (venv) 표시가 보이면 성공입니다. 이걸 꼭 확인하고 개발을 시작하세요. 다 쓴 뒤 가상환경을 끄려면 OS 상관없이 다음을 칩니다.

# 가상환경 비활성화 (끄기)
deactivate

가상환경은 개발 단계뿐 아니라 배포 단계에서도 핵심입니다. 실제 배포는 더 엄격해서, 그 프로젝트가 쓰는 라이브러리 묶음을 그대로 서버로 옮겨 전달해야 하거든요. 그래서 처음부터 가상환경 안에서 작업하는 습관이 개발 전체에 아주 중요합니다. 핵심 한 줄: (venv)가 보이면 성공. 가상환경은 반드시 활성화한 뒤 사용하세요.

5. 코드는 어디서 작성하나요? — IDE

이제 가상환경까지 만들었으니, 코드를 어디에 쓸지 정할 차례입니다. "메모장에서 확장자만 .py로 바꾼 파일을 만들어도 되나요?" 결론부터 말하면 됩니다. 메모장에 쓴 텍스트라도 확장자가 .py면 Python으로 실행할 수 있습니다. 하지만 메모장은 글자를 기록하는 용도일 뿐이라, 그걸로 개발하면 엄청나게 불편합니다. (자동 완성도 없고, 오류 표시도, 문법 색칠도 없죠.)

그래서 개발에 특화된 편리한 도구를 씁니다. 이걸 IDE(통합 개발 환경)라고 합니다. 종류는 아주 많지만, 대표적으로 두 가지를 비교해 봅시다.

VS Code Cursor
만든 곳 Microsoft (VS Code 기반)
특징 전통적으로 가장 많이 쓰인 무료 편집기. 가볍고 빠름 AI가 내장된 VS Code 기반 IDE
강점 확장 기능으로 무한 확장, Python 확장 설치하면 강력 AI가 코드 작성을 적극적으로 도와줌
비용 무료 월 무료 사용량 있음
분위기 대부분의 개발자가 사용 코딩하면서 옆에서 ChatGPT가 돕는 느낌, 인기 급상승

VS Code는 마이크로소프트가 만든 무료 편집기로 오랫동안 개발자들의 사랑을 받아 왔습니다. 그런데 근래에 큰 변화가 일어났죠. AI가 내장된 Cursor가 등장하면서, 옆에서 AI가 코드 작성을 적극적으로 거들어 주는 시대가 됐습니다. 무료 사용량도 기본으로 주고 인기가 빠르게 오르고 있어서, 우리 수업에서는 Cursor를 중심으로 진행하겠습니다.

6. Cursor 설치하고 써 보기 (데모)

설치는 간단합니다. 검색창에 "Cursor"를 치면 공식 사이트가 나옵니다. 클릭해서 들어가 Download를 누르면 됩니다. 특별한 옵션 없이 기본 설정대로 "다음, 다음"만 눌러도 설치가 끝납니다.

실행은 두 가지 방법이 있습니다. 프로그램을 직접 실행해도 되고, 터미널에서 작업할 폴더로 이동한 뒤 다음을 칠 수도 있습니다.

# 현재 디렉토리(.)를 Cursor로 연다 는 뜻
cursor .

Cursor가 열리면 화면 한쪽에 채팅창이 있습니다. 여기에 "이런 걸 만들고 싶다", "이 에러를 해결하고 싶다"고 자유롭게 말하면 AI가 코드를 만들어 주거나 고쳐 줍니다. 화면 아래쪽에는 터미널 창도 있습니다(안 보이는 분은 메뉴에서 터미널을 열면 됩니다). 우리가 앞에서 쓰던 검은 명령 창과 똑같은 것이고, Python을 실행할 때 바로 여기서 돌리면 됩니다.

간단한 파일을 하나 만들어 실행해 봅시다. "New File"로 새 파일을 만들고 확장자를 .py로 지으면, 파일 아이콘이 Python 마크로 바뀝니다. 코드를 쓰다 보면 Cursor가 다음에 칠 내용을 흐릿한 회색 글씨로 미리 제안합니다. 이게 바로 AI가 함께 개발하고 있다는 신호입니다. 그 제안이 내가 원하던 것이면 Tab 키를 눌러 그대로 반영하면 됩니다.

# hello.py — 가장 기본적인 Python 파일 예시
# (Cursor에서 새 파일을 .py 확장자로 만들고 아래를 입력해 봅니다)
print("Hello, World!")

여기서 자주 하는 실수 하나. 파일 이름 옆(또는 탭)에 까만 동그라미가 보이면 "내용은 썼지만 아직 저장이 안 됐다"는 뜻입니다. Ctrl + S로 저장해야 까만 동그라미가 사라지고 변경이 반영됩니다. 저장한 코드를 실행하려면 터미널을 하나 열고 다음처럼 칩니다.

# 작성한 파이썬 파일을 실행한다  (python 다음에 실행할 파일명)
python hello.py

엔터를 치면 우리가 쓴 코드대로 동작하는 결과(Hello, World!)가 출력됩니다.

"말로 코딩하기" — 두 가지 형태

개발이 어렵고 진입 장벽이 높다고 걱정하는 분이 많은데, 지금은 "무엇을 만들지 구체적으로 말할 수만 있으면" 도구들이 정말 잘 받아 줍니다. Cursor에서 AI에게 일을 시키는 방법은 크게 두 가지입니다.

  • (1) 채팅창에서 전체 작업 요청 — 채팅창에 "계산기 프로그램 만들어 줘" 같이 요청하면, AI가 생각한 뒤 calculator.py 같은 파일을 새로 만들고, 실행 방법과 어떻게 만들었는지까지 설명해 줍니다. 웹에서 ChatGPT를 쓸 때처럼 코드를 복사해 붙여넣을 필요 없이, Cursor 안에서 곧바로 반영됩니다. 결과가 마음에 들면 "승인"을 누르면 적용되고, 앞에서처럼 python calculator.py로 실행하면 계산기가 뜹니다.
  • (2) 특정 영역만 콕 집어 수정 — 코드의 일부 영역만 손보고 싶을 때는, 그 부분을 선택하고 Ctrl + K를 누르면 작은 채팅창이 생깁니다. 거기에 "이 부분 이렇게 바꿔 줘"(예: "계산기 크기를 키워 줘")라고 하면 해당 영역만 디테일하게 수정해 줍니다. 마찬가지로 마음에 들면 승인하고, 저장(Ctrl + S)해야 반영됩니다.

즉, 설계(무엇을 만들지)를 명확히 하면, 실제 코드 작성은 이런 도구들이 대신해 줍니다. 그래서 우리가 "어떻게 설계하고, 그 설계가 실제 앱으로 어떻게 만들어지는지"에 시간을 많이 쓴 겁니다. 이 흐름만 익히면 여러분도 정말 많은 것을 만들어 낼 수 있습니다.

7. 한 걸음 더 — 터미널형 AI 코딩 도구 (Claude Code · Gemini CLI)

Cursor도 좋지만, 더 강력한 세계가 있습니다. Claude Code입니다. 이건 앞에서 본 IDE가 아니라 터미널 환경에서 각종 작업을 "에이전트" 형식으로 처리해 주는 AI입니다. 현시점에서 가장 강력하다고 평가받고, 개발 수준도 높으며, 코드 작성 외의 수많은 작업까지 알아서 해냅니다. 다만 단점은 유료 구독이 필요하다는 점입니다. (Claude는 최소 Pro 등급 이상 구독해야 Claude Code를 쓸 수 있습니다.) 그래도 이름은 기억해 두세요 — 현재 개발자들이 인정하는, 성능이 가장 좋은 개발용 AI 에이전트입니다.

다만 우리 수업에서는 무료로 비슷한 경험을 할 수 있는 Gemini CLI를 써 보겠습니다. Claude Code와 여러 면에서 비슷하고, 무료입니다. 성능은 조금 부족한 면이 있지만, 바뀌고 있는 개발 형태를 연습하고 익히기에는 더없이 좋은 선택입니다. (CLI는 Command Line Interface, 즉 "명령줄로 다루는 도구"라는 뜻입니다.)

Gemini CLI는 설치 전에 준비물이 하나 있습니다. Node.js를 먼저 설치해야 합니다. 검색창에 "Node.js"를 쳐서 공식 사이트에 들어가 내 OS에 맞는 설치 파일을 받아 설치하면, 터미널에서 node와 npm 명령을 쓸 수 있게 됩니다. 그다음 설치 명령을 넣으면 됩니다.

# 1단계: Node.js 설치 후, 터미널에서 node / npm 명령이 되는지 확인
node --version
npm --version

# 2단계: Gemini CLI 설치 (npm으로 전역 설치)
npm install -g @google/gemini-cli

# 3단계: 터미널에서 실행 — 'gemini' 라고 치면 대화형 환경이 뜸
gemini

gemini를 실행하면 터미널 안에서 대화 환경이 열립니다. 여기에 "이런 거 개발하고 싶은데 어떻게 해 줘"라고 말로 요청하면 만들어 줍니다. 우리는 앱을 개발할 때 부분적으로 이 도구를 활용할 예정입니다. 현시점에서 가장 선진화된 개발 형태라고 할 수 있습니다.

🎮 게임 기획자 시점

로컬 환경 세팅은 단순한 IT 잡무처럼 보이지만, 사실 "실험"에서 "서비스"로 넘어가는 분기점입니다. Colab 노트북 한 개로는 한 번 분석하고 끝이지만, 로컬에서 여러 파일로 구조를 짜면 "리뷰 수집 → 감성 분석 → 트렌드 집계 → 대시보드"가 반복 가능한 파이프라인이 됩니다. 라이브 게임에서 매주 패치마다 유저 반응을 새로 분석해야 한다면, 바로 이 구조가 있어야 매번 손으로 다시 하지 않고 돌릴 수 있죠. 가상환경은 "이 분석 프로젝트의 환경을 그대로 동료나 서버에 넘겨도 똑같이 돌아간다"는 재현성을 보장하는데, 이건 기획 문서를 누가 읽어도 같은 결론에 닿게 만드는 것과 같은 가치입니다. 그리고 Cursor·Gemini CLI 같은 도구 덕분에, 코딩을 깊이 몰라도 "무엇을 분석할지·어떤 대시보드가 필요한지"를 명확히 말로 설계할 수 있는 기획자가 실제 결과물까지 만들어 낼 수 있게 됐습니다.

⚠️ 흔한 오해 / 짚고 갈 점

  • "최신 버전이 항상 좋다"는 오해. 개발에서는 안정성이 최신보다 중요합니다. 라이브러리 호환성 때문에 한두 단계 아래의 검증된 버전(여기서는 Python 3.12)을 씁니다.
  • "Add Python to PATH" 체크를 빠뜨리는 실수. 이걸 안 하면 터미널에서 python 명령이 안 먹힙니다. 빠뜨렸다면 설치 파일을 다시 실행해 체크하고 재설치하세요.
  • 가상환경을 "만들기만" 하는 실수. python -m venv venv로 만들었다고 끝이 아닙니다. 활성화((venv) 표시 확인) 해야 적용됩니다. 활성화 안 하면 전역 Python을 쓰게 돼 충돌이 다시 생깁니다.
  • 저장 안 하고 실행하는 실수. Cursor에서 코드를 써도 까만 동그라미가 있으면 저장 전입니다. Ctrl + S로 저장해야 실행에 반영됩니다.
  • AI 코딩 도구가 설계까지 대신해 주진 않습니다. "무엇을 만들지"를 구체적으로 말해 줘야 잘 만들어 줍니다. 설계는 여전히 사람의 몫입니다.

✅ 30초 정리

  1. 앱을 만들고 배포하려면 Colab으로는 한계(세션 90분·파일 구조·온라인 필수·배포 불가)가 있습니다. 그래서 내 컴퓨터(로컬)로 옮깁니다.
  2. Python 3.12 안정 버전을 설치하고 "Add Python to PATH"를 꼭 체크합니다. 그리고 프로젝트마다 가상환경(venv)을 만들고 활성화해서 작업하는 게 기본입니다.
  3. IDE(VS Code·Cursor)와 AI 코딩 도구(Claude Code·Gemini CLI)로 효율적으로, 심지어 "말로" 개발할 수 있습니다.
